## Confirmed findings (verification-backed)

### Critical

**C1. `hibernate_sequences` rename without value rescaling risks PK collisions (~40 TABLE entities).** [liquibase-db, also flagged in core-model-entities]
`changeLog-rsdev-444.xml:7` renames `sequence_next_hi_value` to `next_val`. On `main`, `hibernate.cfg.xml:19` set `new_generator_mappings=false`, so `@GeneratedValue(strategy=TABLE)` entities used Hibernate 5's legacy hi/lo generator, whose stored value is a hi-counter (real next id is a multiple of it), not the next id. Hibernate 6's enhanced `TableGenerator` reads `next_val` as the literal next id. The column rename is a red herring: the real issue is the generator-algorithm change reinterpreting existing values, which can hand out ids at or below existing PKs. **This directly refutes `spring6-todo.md:39` ("reviewed, sound, effectively a no-op").** Verified by 2 independent verifiers (both rated critical). See B1 for the fix.

**C2. core-model `HibernateSearchTest` queries stale field names (4 assertions cannot pass).** [core-model-search]
After the `fields.fieldData` field was split into `extraFields.fieldData` / `files.fieldData` / `notes.fieldData`, the test still issues `f.match().field("fields.fieldData")` for extra-field, attachment-name and note-content cases. `fields.fieldData` now carries only `InventoryEntityField` content. This is CI breakage, not a production fault (verifiers suggested downgrading to high), but the core-model branch suite will not go green until fixed. See B6.

**C3. core-model test bootstrap registers no `LuceneAnalysisConfigurer`.** [core-model-search]
`HibernateUtils.createConfiguration()` sets the Lucene backend but never sets `hibernate.search.backend.analysis.configurer`. Indexed entities reference `structureAnalyzer` / `aclAnalyzer`, so HS7 strict bootstrap fails and the search tests cannot start. This contradicts core-model review C1 ("no action needed in this repo"): the configurer is needed by the core-model test harness too. See B6.

### Medium (confirmed)

- **M1. REVERTED (`3309c75d5`, 2026-06-10; original change `d7b424cac` same day). The explicit advisors are load-bearing at runtime; `@Transactional` alone is NOT reliable for service beans.** The original M1 removed the `contentInitializerTx`, `shareApiTx`, `contentUpdaterTx`, `archiveExportPlannerTx` and `internalFileStoreTx` advisors in favour of class-level `@Transactional`. This broke first login in the running app (500 on /workspace: "Could not obtain transaction-synchronized Session" in `ContentInitializer.init`, then `LazyInitializationException` in `UserContentUpdaterImpl`). Mechanism, proven via TRACE on `org.springframework.aop`/`.transaction`: `shiroFilter` (`ShiroFilterFactoryBean`, security.xml) is a BeanPostProcessor, so its `securityManager` reference eagerly instantiates the realm's entire service-bean dependency graph (~200 beans, including `contentInitializer` and `userContentUpdater`) during BPP registration; in that window Spring's advisor retrieval logs "Skipping currently created advisor 'org.springframework.transaction.config.internalTransactionAdvisor'", so those beans are proxied permanently without transaction advice (SHIRO-829). The XML pointcut advisors finish creating instantly and still apply inside the window, which is why `*Manager` beans kept working. The original M1 verification tests could not catch this: the test context (`applicationContext-test-service.xml`) still carries the old advisors, never loads security.xml, and transactional test bases supply their own session. The readOnly-discarded-by-double-advice observation remains true but is main-parity behaviour; fixing it requires first fixing the early-instantiation cascade (see follow-up below). Verified after revert: fresh first login (user2b) initializes content and renders /workspace; DBDataIntegrityCheckerTest, UserContentUpdaterImplTest, ShareApiServiceTest green.
- **M1a. FOLLOW-UP (new, 2026-06-10). ## B1/B2 design decisions (2026-06-04)

Both decisions were taken with Fraser and implemented + validated against a full copy of the local dev DB (a representative pre-upgrade Hibernate 5 database: 23 per-entity `hibernate_sequences` rows, legacy `sequence_next_hi_value` column, BATCH tables on the Batch 3 layout). The scratch copies were dropped after validation; the real `rspace` DB was not modified.

### B1 — keep per-entity sequences (option "A1"), do not consolidate

**Problem (refined during implementation).** Two distinct failure modes, not one:
1. *Value reinterpretation.* Hibernate 5 (`new_generator_mappings=false`) used the legacy `MultipleHiLoPerTableGenerator`, whose stored value is a hi/lo marker, not the next id. The local DB proved this starkly: `Field`, `FormUserMenu` and the `Communication` hierarchy store a counter of `1`/`2` but their real `MAX(id)` is ~32768 (`Short.MAX_VALUE`, the legacy `maxLo`). Hibernate 6's enhanced `TableGenerator` reads the value far more literally, so it would hand out ids at/below existing PKs.
2. *Segment relocation.* Hibernate 6's enhanced `TableGenerator` defaults bare `@GeneratedValue(strategy=TABLE)` entities to a single shared `default` segment row (confirmed in the 6.4.4 `TableGenerator` bytecode: `DEF_SEGMENT_VALUE="default"`, `determineDefaultSegmentValue` reads `prefer_entity_table_as_segment_value`). Existing DBs only have per-entity rows and no `default` row, so on first insert Hibernate creates `default` at 1 and collides across ~47 tables. The original review (and the `spring6-todo.md` "sound, effectively a no-op" note) missed this second mode.

**Options considered.** (1) Keep Hibernate's default and consolidate the 38 bare entities onto one shared `default` counter; simplest changeset but a behaviour change (interleaved/sparse ids, dead legacy rows). (2) Preserve the current per-entity counters. (3) Switch to `IDENTITY`/auto-increment — rejected: the two `TABLE_PER_CLASS` hierarchies require ids unique across sibling tables, which per-table auto-increment cannot provide, and `IDENTITY` disables Hibernate insert batching; out of scope for the migration.

**Decision: option 2 (preserve current per-entity behaviour), implemented as "A1" (config, low churn).** *[Corrected by B1a, 2026-06-06: the setting is NOT honoured as a global hibernate property, so the effective behaviour is option 1 (bare entities consolidate onto the shared `default` segment); safety rests on the insurance row. The dead config line was removed from both `applicationContext-dao.xml` files on 2026-06-10.]* Set `hibernate.id.generator.prefer_entity_table_as_segment_value=true` in main + test `applicationContext-dao.xml` so Hibernate 6 keeps naming each entity's segment after its table (matching the existing rows), rather than the 38-explicit-`@TableGenerator` route ("A2") which has the same correctness dependency (table name == legacy row name) but more churn. The reseed changeset additionally seeds a `default` insurance row above the global max, so even if the setting is ever not honoured the fallback path is still collision-safe — making safety independent of the config-key behaviour. The `allocationSize` inconsistency (M7: two pinned entities at 1, the rest defaulting to 50) was left as-is; the reseed margin of `+51` (a full default block + 1) is safe for the pooled, pooled-lo and no-op optimizers regardless.

**Changeset.** `hibernate6-reseed-sequences` (`changeLog-rsdev-444.xml`, `context="run"`, guarded on `columnExists next_val`, MARK_RAN otherwise). A self-contained stored procedure loops over **every** existing `hibernate_sequences` row and, for any row whose name is a real base table with an `id` column, sets `next_val = MAX(id)+51`. Driving off the existing rows makes it complete by construction (a table only has a row if it has data, so nothing at-risk can be missed — this is how `StoichiometryInventoryLink` and the `InventoryRecord`/`MappedSuperclass` subclass tables are covered without enumerating them). The two `TABLE_PER_CLASS` segments are handled explicitly. Requires `CREATE/DROP ROUTINE` privilege at migration time (the app schema user has `ALL` in the standard deployment; this is the first stored-procedure changeset in the project).

**Owed before merge:** ~~the empirical segment-name check~~ DONE 2026-06-06 (see B1a below).

### B1a. `prefer_entity_table_as_segment_value` is NOT honoured (VERIFIED 2026-06-06)

The populated-DB boot check passed for safety, but it disproved the central assumption of the A1 decision above. Booting the Spring 6 app against the populated local `rspace` DB and logging in (which inserts a `UserAccountEvent`, a bare `@GeneratedValue(strategy=TABLE)` entity) showed:

- The `default` segment advanced (71660 to 71760), while `UserAccountEvent`'s own per-entity row stayed frozen at 55, and the new row's id (71612) came from a `default` pooled block.
- So bare TABLE entities draw ids from the shared `default` segment, NOT their table-named per-entity row. With the A1 setting working as the doc assumed, the opposite would happen.

Root cause: `prefer_entity_table_as_segment_value` is a `TableGenerator`-LOCAL config parameter (`org.hibernate.id.enhanced.TableGenerator.CONFIG_PREFER_SEGMENT_PER_ENTITY`), not a global `hibernate.*` setting. It is absent from `org.hibernate.cfg.AvailableSettings` in both `hibernate-core` 6.2.41 and 6.4.4 (checked in the `.m2` jars). Setting it as a global hibernate property in `applicationContext-dao.xml` does not propagate it to the implicit generators, so Hibernate falls back to the single `default` segment for every bare TABLE entity. Only `AbstractUserOrGroupImpl` and `Communication` (which carry an explicit `@TableGenerator(...pkColumnValue=...)`) use a named segment.

Consequences:
- **Safety is intact.** Collision safety comes entirely from the `default` insurance row seeded above the global max, exactly the fallback the changeset's own comment anticipates ("even that fallback path cannot collide"). The live test confirmed no new row and no collision. B1 is closed.
- **The per-entity reseed is vestigial** for the 38 bare entities (harmless belt-and-braces). Only the two explicit-`@TableGenerator` segments are actually read.
- **Two doc statements are now false** and should be corrected: (1) `changeLog-rsdev-444.xml:52-53` ("With the setting on ... Hibernate reads the per-entity rows instead and the `default` row is simply unused") — the reverse is true; and (2) the A1 decision text below claiming the setting "keeps naming each entity's segment after its table". The accurate description is that the migration consolidates bare entities onto `default` (the considered "option 1" behaviour) and relies on the insurance row.
- **Follow-up decision — DONE (2026-06-10):** removed the dead `prefer_entity_table_as_segment_value` line from both main and test `applicationContext-dao.xml`, rewrote the surrounding comment and the two false statements in the `changeLog-rsdev-444.xml` changeset comment to describe the verified `default`-consolidation behaviour, and corrected the A1 decision text above.

Separately flagged: both `hibernate-core` 6.2.41.Final and 6.4.4.Final are present in `.m2`; the tracker cites 6.4.4. Confirm the effective resolved version (does not affect the B1 conclusion, since the parameter is absent as a global setting in both).

### B2 — drop + recreate the BATCH_* tables (chosen over a hand-written ALTER)

**Decision.** On an un-migrated existing DB, drop and recreate the `BATCH_*` tables from the Batch 5.2.1 jar's own `schema-drop-mysql.sql` + `schema-mysql.sql` (`hibernate6-batch5-schema-migration`, guarded on `BATCH_JOB_EXECUTION` present AND old `TYPE_CD` column present). Chosen over a surgical ALTER because the 3→5 jump is large (notably the full `BATCH_JOB_EXECUTION_PARAMS` redesign) and these are async-export monitoring tables, so discarding job-execution history on upgrade is acceptable. Version-accurate by construction (reads whatever Batch version is on the classpath) and idempotent (no-op once `TYPE_CD` is gone; no-op on fresh installs already built by `changeLog-1.47.xml::17-09-10z`).

**Blast radius (traced through the code, 2026-06-04).** These tables back ONLY the REST API v1 export (`/api/v1/export` launched via `JobLauncher` in `ExportApiSpringBatchHandlerImpl`; status polled via `JobExplorer.getJobExecution(jobId)` in `JobsApiHandlerImpl`). The interactive web UI export is a separate mechanism (`ExportController` to `ExportImport.asyncExport...`, the plain `@Async` path) and never touches `BATCH_*`. There is no FK from any RSpace table into `BATCH_*`, and no job-history listing (only point-in-time `getJob(jobId)`). The archive file itself lives on the filestore and is downloaded **by filename**, not by jobId: `GET /export/ajax/downloadArchive/{zipname}` (`ExportImportImpl.streamArchiveDownload`) opens `new File(exportFolderLocation, zipname)` and streams it, consulting no DB at all (not `BATCH_*`, not `ArchivalCheckSum`). So dropping the batch tables does NOT make any archive undownloadable; the only thing lost is the jobId-to-result lookup for an API job whose result a client had not yet fetched when the server was upgraded (remedy: re-export). In-flight jobs die from the JVM restart regardless of the drop (the export tasklet is in-memory, not restart/resume-wired).

**Refinement implemented: preserve jobId continuity.** Spring Batch allocates ids from the `BATCH_*_SEQ` incrementer tables (next id = stored `ID` + 1). A plain drop+recreate reseeds them to 0, so new jobIds restart near 1 and a counter climb could eventually re-issue a pre-upgrade jobId (cross-user access is blocked by the `export.user` permission check, but same-user reuse could confuse a stale poll). To remove this entirely the changeset captures `MAX(JOB_EXECUTION_ID)` / `MAX(JOB_INSTANCE_ID)` / `MAX(STEP_EXECUTION_ID)` into a temporary holding table BEFORE the drop, then after recreate seeds `BATCH_JOB_EXECUTION_SEQ` / `BATCH_JOB_SEQ` / `BATCH_STEP_EXECUTION_SEQ` to those maxima so new ids continue monotonically. The holding table survives the drop (it is not a `BATCH_*` table) and is removed at the end. Validated on a copy of the local dev DB with simulated high ids: the incrementer hands out `max + 1` next.
